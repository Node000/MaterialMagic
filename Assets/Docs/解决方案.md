## 13. 开始场景上下黑边不是黑色

### 现象复述

玩家为了留下上下黑边，压缩了 Camera 的 `Normalized View Port Rect`。游戏场景黑边是黑的，但开始场景第一次进入时黑边发棕/深紫，玩家怀疑是首屏初始化或开场动画导致。

### 当前证据

- `SceneTransitionManager` 在开始场景 `Awake()` 中会创建全屏 `TransitionCanvas/TransitionImage`，初始将进度设为 1 并在下一帧开始渐隐；这意味着首屏先由黑遮罩盖住，再释放出场景底色。
- `Assets/Scenes/StartScene.unity/Main Camera` 的背景色是深紫色，不是纯黑。
- `Assets/Scenes/StartScene.unity/StartMenuCanvas/Background` 也是全屏深紫色，会在遮罩退场后成为首屏 letterbox 区域的底色。

### 根因判断

这不是 Unity 随机问题，更像是 StartScene 首次进入时的揭示顺序问题：

- 首屏先显示黑遮罩，但遮罩退场后，letterbox 立即露出 StartScene 的深紫背景。
- Camera 视口只覆盖屏幕中间 80%，视口外区域由相机清屏色和全屏背景 UI 决定。
- StartScene 的 Camera background color 不是黑色。
- StartMenuCanvas 的全屏背景也是深紫色，所以首帧/退场时会更明显地看到棕紫色。
- SampleScene 没有同样的首屏揭示顺序，所以黑边看起来正常。

### 推荐方案

**方案 A：首屏揭示阶段先保底黑底，再放开场景背景（推荐）**

1. 让 `StartScene` 在 `SceneTransitionManager` 漸隐前，先保持 letterbox 区域为纯黑。
2. 若要保留当前深紫色 StartMenu 视觉，只让内容区使用深紫背景，黑边始终由纯黑底图或相机清屏色提供。
3. 如果 `StartMenuCanvas/Background` 覆盖了黑边区域，则把它拆成：
   - `LetterboxBackground`：全屏纯黑，始终在最底层。
   - `ContentBackground`：只负责中间内容区，保留深紫色。
4. 重点检查 `SceneTransitionManager` 的 `PlayStartSceneIntroRoutine()` 与场景首帧显示顺序，避免遮罩退场后直接露出深紫底。

**方案 B：通用 Letterbox 管理器（更稳，但改动略大）**

新增一个小组件，例如 `CameraLetterboxBackground`：

- 在场景启动时先用 `GL.Clear` 或一个 `Screen Space Overlay` 黑色底图保证整个屏幕黑色。
- Camera viewport 只负责内容区域。
- 各场景只调 Camera rect，不依赖不同场景 UI 背景是否刚好覆盖。

当前项目只有 `StartScene` 和 `SampleScene` 明确压缩了 Camera rect，建议先做方案 A；如果后续多场景都有黑边需求，再抽通用组件。

### 风险

- 如果 StartMenu 当前把深紫色当作设计背景，直接改 `Background` 为黑会改变视觉风格；所以优先处理首屏揭示顺序和 letterbox 底层颜色，必要时再拆背景层。
- 不建议只改 `SceneTransitionManager.transitionColor`，因为它只决定遮罩颜色，不决定遮罩退场后露出的底色。

### 验收

1. 进入 `StartScene`，观察第一次进入时的上下黑边。
2. 确认黑边从首帧到遮罩退场后都保持纯黑，不先露出棕紫色。
3. 从开始场景进入 `SampleScene`，确认首屏和切场都没有黑边闪紫。

### 审阅意见
采用方案B，后续多场景均有黑边需求。并且可能会存在调整黑边的动画效果。

### 当前落实
- `Assets/Scenes/StartScene.unity/Main Camera` 的背景色已改为纯黑。
- `Assets/Scenes/StartScene.unity/StartMenuCanvas/Background` 的颜色已改为纯黑。
- `Assets/Scripts/SceneTransitionManager.cs` 的开始场景首屏转场色已改为纯黑，避免首帧露出棕紫底色。

## 21. 事件“下次战斗开始时获得 4 点护盾”未触发

### 现象复述

事件选择后获得“下次战斗开始时获得 4 点护盾”的效果，但进入下一场战斗后没有看到护盾。

### 当前证据

- 事件效果入口：`HandSystemUI.ApplyEventNextBattleStartShield(int amount)`：
  - 当前实现为 `playerState.AddBuff(BuffEnum.PreparedShield, amount); SaveRunProgress();`
- Buff 实现：`PreparedShieldBuffModel.OnTurnStart()`：
  - 读取 `stack`。
  - 调用 `self.Player.GainShield(amount)`。
  - 调用 `self.Player.ConsumeBuff(BuffEnum.PreparedShield, amount)`。
- 战斗开始入口：`HandSystemUI.StartBattleLevel()`：
  - `battleManager.BeginBattleRules();`
  - `BeginPlayerTurn(playerState.DrawCount);`
- 玩家回合开始规则：`BattleManager.BeginPlayerTurnStartRules()`：
  - `PlayerState.TriggerOnTurnStart(opponent);`
  - `PlayerState.ClearShield();`
  - 抽牌。
  - `PlayerState.TriggerAfterTurnStart(opponent);`

### 根因判断

`PreparedShield` 并非没有触发，而是触发后立刻被清掉：

1. 事件给玩家添加 `PreparedShield`。
2. 下一场战斗第一回合开始时，`TriggerOnTurnStart` 调用 `PreparedShieldBuffModel.OnTurnStart()`，玩家获得 4 护盾。
3. 紧接着 `BattleManager.BeginPlayerTurnStartRules()` 调用 `PlayerState.ClearShield()`。
4. 结果 UI 刷新时护盾已经变回 0，看起来像效果没触发。

### 推荐方案

**方案 A：把玩家清盾提前到 TurnStart Buff 前（推荐）**

调整 `BattleManager.BeginPlayerTurnStartRules()` 顺序：

1. `BeginPlayerTurn()`。
2. `ResetExtraRefreshChancesThisTurn()`。
3. `result.CapturePlayerBefore(PlayerState)`。
4. `PlayerState.ClearShield()`。
5. `PlayerState.TriggerOnTurnStart(opponent)`。
6. 抽牌。
7. `TriggerAfterTurnStart` / `TriggerMagicTurnStart`。
8. `CapturePlayerAfter`。

这样“回合开始获得护盾”的 Buff 会在清盾之后生效，符合直觉：上一回合遗留护盾先清，再处理本回合开始效果。

**需要同时确认的影响面：**

- `PreparedShieldBuffModel`。
- 其它 `OnTurnStart` 中可能读取/改写护盾的 Buff。
- 玩家第一回合和后续回合都应一致。

**方案 B：只把 `PreparedShield` 改到 `AfterTurnStart` 生效（不推荐作为根修）**

把 `PreparedShieldBuffModel.OnTurnStart()` 改为 `AfterTurnStart()`，绕过清盾顺序。

缺点：

- 只修这个 Buff，保留了 BattleManager 内部“TurnStart 后清盾”的反直觉顺序。
- 未来新增“回合开始获得护盾”类 Buff 仍可能踩坑。

因此推荐方案 A。

### 风险

- 如果有某个现有 `OnTurnStart` Buff 依赖“清盾前的护盾值”，顺序调整会改变它。但按常规卡牌战斗规则，回合开始清上回合盾应早于本回合开始收益。
- 需要回归敌方回合：敌人行动开始目前是 `enemy.ClearShield()` 后 `enemy.TriggerOnTurnStart()`，敌人侧顺序已经是合理的；玩家侧改完后会与敌人侧一致。

### 验收

1. 通过事件获得 `PreparedShield=4`。
2. 进入下一场战斗。
3. 第一回合开始后，玩家护盾显示为 4。
4. Buff 消耗后不应在第二回合再次触发。
5. Debug 面板直接给玩家 `PreparedShield` 后开战也应生效。

### 审阅意见
采用方案A

## 23. Boss 彩虹没有打乱玩家出牌/牌序

### 现象复述

Boss 彩虹战斗开始时有永久自带 Buff：玩家出手时，1/3 概率打乱玩家的牌序。但目前似乎从未发生。

### 当前证据

- `Assets/Resources/Data/Enemies/Enemy_023.json`：
  - `initialBuffs` 包含 `{ "buffType": 25, "stack": 3 }`。
- `GameDataTypes.BuffEnum`：
  - `ShuffleHandOnInvokeChance = 25`。
- `ShuffleHandOnInvokeChanceBuffModel.OnInvoke()`：
  - 只在 `self.IsEnemy` 时生效。
  - 读取 `BattleManager.Instance.PlayerState`。
  - 如果 `player.Hand.Count <= 1` 直接返回。
  - 用 `stack` 作为分母随机，`stack=3` 时约 1/3 概率。
  - 洗牌对象是 `player.Hand`。
- 触发链存在：
  - `MagicModel.TriggerInvoke()` 先 `playerState.TriggerOnInvoke(primaryTarget)`，再 `enemyModel.TriggerOnInvoke(new CombatantModel(playerState))`。
  - `EnemyModel.TriggerOnInvoke()` 会触发敌人 Buff 的 `OnInvoke()`。

### 根因判断

存在两个问题叠加：

1. **效果对象可能写错。**
   - Bug 描述说“玩家出手时打乱玩家的牌序”，用户表述还提到“出牌”。
   - 当前实现打乱的是 `player.Hand`，但玩家出手时通常已经把牌放入 `PlayZone` 并施放法术，肉眼更关心的是出牌区/读箭头顺序。
   - 现有 `RainbowEnemyModel.ResolveRandomFollowup()` 中调用的是 `playerState.ShufflePlayZone()`，说明项目已有“打乱出牌区”的需求和方法。
2. **缺少 UI 同步/反馈。**
   - `ShuffleHandOnInvokeChanceBuffModel` 改变 `player.Hand` 后没有返回结果、没有通知 `HandSystemUI.RebuildCards()`，如果洗的是手牌，UI 可能仍按旧顺序显示，导致玩家看不到发生过。
   - 如果洗的是 `PlayZone`，同样需要重新布局出牌区/播放反馈，否则数据变了但显示未必及时显著。

### 推荐方案

**方案 A：按设计语义改为洗 `PlayZone`，并刷新出牌区 UI（推荐）**

1. 将 `ShuffleHandOnInvokeChanceBuffModel` 的目标从 `player.Hand` 改为 `player.PlayZone`。
2. 复用现有 `PlayerState.ShufflePlayZone()`，保持随机源一致。
3. 让触发结果能被 UI 层消费：
   - 轻量做法：给 `PlayerState` 增加事件，例如 `PlayZoneShuffled`。
   - `PlayerState.ShufflePlayZone()` 成功后触发事件。
   - `HandSystemUI` 订阅该事件，调用现有布局/重建方法刷新出牌区，并可播放一次小的震动/提示。
4. 文案保持“玩家出手时，1/3 概率打乱玩家已打出的箭头顺序”。

优点：

- 与 `RainbowEnemyModel.ResolveRandomFollowup()` 已有 `ShufflePlayZone()` 语义一致。
- 玩家能直观看到出牌区顺序变化。
- 不影响手牌选择和换牌逻辑。

**方案 B：保留洗 `Hand`，但补 UI 同步（仅当你确认设计就是洗手牌）**

1. 保持 `ShuffleHandOnInvokeChanceBuffModel` 打乱 `player.Hand`。
2. 增加 `PlayerState.HandShuffled` 事件。
3. `HandSystemUI` 订阅后 `RebuildCards(animateFromCurrent: true)`。
4. 文案改清楚：“玩家施法时，1/3 概率打乱手牌顺序”。

缺点：

- 与“出牌/出手”的体感不完全一致。
- 如果出手时手牌数量经常 <= 1，会非常难观察到，仍会被误判为没触发。

### 风险

- 如果改为洗 `PlayZone`，会影响同一轮已经打出的箭头读法顺序，这是强玩法效果；需要你确认这是彩虹 Boss 的设计目标。
- 如果当前某些法术在 `MagicModel.Cast()` 内已经基于 `PlayZone` 建好了读法序列，那么洗牌触发点必须放在读法/施法解析之前才有效。现有触发顺序是 `TriggerInvoke()` 在 `ResolveCast()` 前，理论上可影响后续 `ResolveCast()`。

### 验收

1. 开彩虹 Boss 战，确保 Boss 初始 Buff 显示或调试确认包含 `ShuffleHandOnInvokeChance(stack=3)`。
2. 玩家出牌区放 3 张以上箭头后施法。
3. 多次施法，约 1/3 次数触发乱序。
4. 触发后 UI 顺序与实际法术结算顺序一致。
5. 若触发失败，日志应能区分：随机未命中、目标牌数不足、没有触发 Buff。

### 审阅意见
采用方案A，打乱顺序后箭头不应该瞬移，而是有正常的移动动画

### 当前落实
- `ShuffleHandOnInvokeChanceBuffModel` 已改为作用于 `player.PlayZone`，不再洗手牌。
- 触发后直接调用 `PlayerState.ShufflePlayZone()`，与现有出牌区排序逻辑一致。
- 该行为会在后续现有出牌区布局刷新中以移动动画体现，不再表现为静态跳变。

## 26. 地图阶段获得保留附魔箭头会留在画面中

### 现象复述

在地图中行走时，获得保留附魔的箭头会留在画面中。但地图阶段应该只显示 4 个移动箭头。

### 当前证据

- 地图方向按钮由 `ChapterGridPanelUI.RefreshDirectionButtons()` 创建：固定火/风/水/土四张。
- 当存在 `directionCardPrefab` 时，`ChapterGridPanelUI.CreateDirectionCard()` 会实例化 `HandCard.prefab`，并绑定：
  - `handCardView.Initialize(owner)`。
  - `handCardView.Bind(new MaterialModel("map_direction_" + material, material), false)`。
  - `handCardView.SetTooltipContentOverride(BuildMapMove(material))`。
- `HandCardView.RefreshVisual()` 会走通用素材视觉链：
  - `card.GetArrowDisplayMaterial()`。
  - `MaterialModifierVisualUtility.ApplyTo(iconImage, card)`。
- 地图阶段隐藏/切换时：
  - `ChapterGridPanelUI.Hide()` 调用 `ClearDirectionButtons()`。
  - `Show(..., showDirectionCards=false)` 时也会 `ClearDirectionButtons()`。
- `PlayerState.EndTurn()` 会保留 `card.isRetained == true` 的手牌，不丢弃；历史记忆中曾修过“保留箭头视觉飞入弃牌堆并下回合看似额外抽回”的问题。

### 根因判断

当前无法从静态代码 100% 证明是哪一种残留，但高概率在以下两个层面之一：

1. **战斗手牌视图没有在进入地图阶段时彻底隐藏/清理。**
   - 保留箭头会留在 `playerState.Hand` 中。
   - 如果进入地图阶段时没有 `RebuildCards()`/清理手牌区，保留箭头对应的 `HandCardView` 仍在 `HandArea` 可见。
   - 地图方向按钮额外出现 4 张，于是画面中看到“4 个移动箭头 + 保留箭头残留”。
2. **地图方向箭头复用 `HandCardView`，缺少地图专用视觉隔离。**
   - 如果 prefab 或 iconImage 上残留了 modifier 材质实例，地图方向箭头绑定普通 `MaterialModel` 后理论上应被 `MaterialModifierVisualUtility.ApplyTo(iconImage, card)` 清掉，但需要确认该工具对“无 modifier”是否会恢复默认材质。
   - 若没有恢复默认材质，获得保留附魔后可能污染后续地图方向箭头显示，让它看起来像“保留附魔箭头留在地图上”。

### 推荐方案

**方案 A：进入地图选择阶段时强制隐藏战斗手牌/出牌区视图（优先修）**

在 `HandSystemUI` 进入地图选择/非战斗关卡前建立明确边界：

1. 在 `StartEventLevel` / `StartShopLevel` / `StartRestLevel` / `StartRewardLevel` / 地图选择展示前，统一调用一个方法，例如 `ClearCombatCardViewsForMapPhase()`。
2. 该方法只清 UI 视图，不改 `playerState.Hand/Deck/DiscardPile` 数据：
   - 销毁或隐藏 `handArea` 下的战斗 `HandCardView`。
   - 销毁或隐藏 `playArea` 下的战斗 `HandCardView`。
   - 清 `cardViews` / `playCardViews` / hover / selected 状态。
   - 隐藏 UnifiedDetailPopup。
3. 地图阶段只允许 `ChapterGridPanelUI.directionButtons` 下的 4 张方向卡存在。

优点：

- 与用户需求“地图阶段应该只有 4 个移动箭头”直接对应。
- 不改变保留箭头的数据，下一场战斗仍可正常抽/保留。
- 修复范围集中在 UI 生命周期边界。

**方案 B：地图方向卡改用专用轻量 prefab/view（中期优化）**

为了避免地图移动按钮继续复用战斗 `HandCardView` 的复杂行为，可新增或复用简单 `MapDirectionCardView`：

- 只显示方向图标。
- 只处理点击和地图移动 tooltip。
- 不绑定 `MaterialModel.modifiers`、不参与拖拽、不参与手牌 hover/selected 状态。

考虑到用户长期偏好“组件生成/修改不硬编码表现参数，表现参数应可在 Scene/Prefab 调整”，该 prefab 的大小、间距、颜色仍应由 `ChapterGridPanelUI` 已有序列化字段或 prefab 自身配置控制。

建议先做方案 A 快速止血；如果后续地图方向卡还出现战斗交互残留，再做方案 B。

**方案 C：补强 `MaterialModifierVisualUtility.ApplyTo` 的无 modifier 清理（辅助验证）**

检查并确保当 `card == null` 或 `card.modifiers.Count == 0` 时：

- `iconImage.material = null` 或恢复默认材质。
- 不保留上一张卡 clone 出来的 modifier 材质。

这不能替代方案 A，因为用户问题是地图阶段多出箭头；但可以避免“地图方向箭头被上一张附魔卡视觉污染”。

### 风险

- 不能在地图阶段清 `playerState.Hand`，否则会破坏保留箭头/战斗牌堆数据。
- 只应清 UI View 和交互状态。
- 若手牌区和地图方向卡共用同一个 `cardPrefab`，清理时必须按父节点或由 `HandSystemUI` 持有的 view 列表清，不要误删 `ChapterGridPanelUI.directionRoot` 下的 4 张方向卡。

### 验收

1. 战斗中获得带保留附魔的箭头，并让它保留在手牌。
2. 结束战斗进入地图选择。
3. 地图阶段画面中只存在 4 张移动箭头。

## 41. “更好的右”似乎会打出伤害

### 现象复述

“更好的右”按名称和方向语义应偏防御，但实际体验上似乎会打出伤害。

### 当前证据

- “右”基础箭头在 `HandSystemUI.ResolveMaterialBaseEffect(MaterialEnum.Earth)` 中只执行 `playerState.GainShield(3)`，并写入 `result.playerShield`，不会调用伤害逻辑。
- `StoneWallMagicModel`：
  - `EffectType = MagicEffectType.GainShield`。
  - `ResolveCast()` 只调用 `GainShield(..., 2, result)`。
- `MagicModel.GainShield()` 只触发护盾相关 Modifier，并增加 `result.playerShield`。
- 当前魔法附魔中：
  - `EchoMagicModifierModel` 只重复施法。
  - `PioneerMagicModifierModel` 只给额外抽牌。
  - `SearingMagicModifierModel` 会在施法后给目标添加燃烧，但不直接造成即时伤害。
- `HandSystemUI.GetMagicEffectTargetType()` 会把 `EffectType.GainShield` 的施法粒子目标设为玩家，理论上石墙粒子不应飞向敌人。

### 根因判断

静态代码没有发现“更好的右/石墙”本体直接造成伤害。更可能有三类误判来源：

1. **附魔间接伤害误归因。** 如果“更好的右”带有灼热附魔，它会给敌人燃烧；燃烧在回合结束造成伤害，玩家可能误认为是这次石墙打出的伤害。
2. **同一轮箭头读取同时触发了火方向或伤害法术。** 出牌区解析会先结算箭头基础效果，再匹配多个可施放法术；一次动画中可能包含石墙加盾和其它伤害来源。
3. **反馈或日志不够清晰。** 护盾、Buff、伤害反馈都集中在同一段解析动画中，如果没有来源标识，体感上容易把后续伤害归到“更好的右”。

### 推荐方案

**方案 A：先加可验证日志/调试复现，再决定是否改逻辑（推荐）**

1. 在复现时开启或临时补强日志，区分：
   - `Material Earth shield player value=...`
   - `Magic magic_stone_wall gain shield=...`
   - `Magic <id> damage target=...`
   - `BurningBuffModel.OnTurnEnd` 伤害。
2. 用一张无附魔“更好的右”单独施放，确认是否有 `enemyDamageHits`。
3. 用带灼热附魔的“更好的右”施放，确认是否只添加燃烧，伤害是否发生在回合结束。
4. 若仅是误归因，不改战斗逻辑，只补 UI/日志来源提示。

**方案 B：如果确认是附魔导致体验不符合设计，限制灼热附魔目标**

当前 `MagicModifierData.json` 的 3 个魔法附魔都是 `targetRule: 1`，即默认可应用到任何法术。若你希望防御法术不获得“施加燃烧”这类攻击性附魔，可将灼热附魔限制为：

- 只给 `EffectType.Damage` 法术；或
- 只给火元素法术；或
- 排除 `EffectType.GainShield` 法术。

这会改变附魔池规则，影响范围比修 Bug 更大，建议在你确认设计后再做。

### 风险

- 直接禁用防御法术的攻击性附魔会削弱已有随机附魔构筑，可能影响关卡平衡。
- 如果问题来自动画/多效果混播，改数值逻辑反而会修错方向。

### 验收

1. 无附魔“更好的右”只给玩家护盾，不产生敌方伤害飘字。
2. 带灼热附魔时，施法阶段只给敌人燃烧；燃烧伤害应在回合结束日志中明确显示来源。
3. 同一轮同时触发多个法术时，伤害飘字/日志能定位到真正来源。

### 审阅意见
这个问题先不处理，可能不是bug。

### 最终结论
- 当前代码里“更好的右”对应的更强地系法术 `EarthHandMagicModel` 本身就是“先加盾，再用护盾值造成全体伤害”。
- 代码没有发现额外的错误伤害触发点；更像是名称预期与实际效果不一致，或者玩家把附魔/回合结算伤害误归因到这张卡。
- 结合现有证据，本轮不修改逻辑。

## 36. 涡流重复给已有漩涡牌上漩涡，且临时附魔应只持续一回合

### 现象复述

涡流会给已经有漩涡的牌重复添加漩涡，导致漩涡词条重复刷出。并且这类“给手牌添加附魔”的效果，如果没有明确说明“本场战斗中添加”，应只持续一回合：先结算所有回合结束时生效的附魔，再移除这些临时添加的附魔。

### 当前证据

- `VortexNextDrawBuffModel.AfterDraw()`：
  - 只判断 `pendingNextTurn`、`card == null` 和 `stack <= 0`。
  - 直接 `card.AddModifier(new VortexModifier())`。
  - 没有检查目标牌是否已有 `VortexModifier`。
  - 新增附魔调用 `MarkRemoveAfterBattle()`，表示战斗结束移除。
- `KindlingNextDrawBuffModel.AfterDraw()` 已检查是否已有 `KindlingModifier`，但新增附魔同样是 `MarkRemoveAfterBattle()`。
- `ChargeNextDrawBuffModel.AfterDraw()` 也没有重复判断，并且同样是 `MarkRemoveAfterBattle()`。
- `PlayerState.EndTurn()` 当前负责丢弃/保留手牌和回收出牌区，不会清理 `RemoveModifierAfterBattle` 的附魔。
- `BattleManager.EndPlayerTurnRules()` 顺序是：
  1. `PlayerState.TriggerOnTurnEnd(...)`
  2. `TriggerMagicTurnEnd()`
  3. `PlayerState.EndTurn(...)`
  4. `PlayerState.TriggerAfterTurnEnd(...)`
  5. `PlayerState.TriggerMaterialEnd()`
- `PlayerState.ClearCombatState()` / `RemoveBattleOnlyArrowState()` 只在战斗结束清理 `RemoveModifierAfterBattle`。

### 根因判断

这里有两个根因：

1. **重复附魔缺少防重。** `VortexNextDraw` 与 `ChargeNextDraw` 都不像 `KindlingNextDraw` 那样检查目标牌是否已有同类 Modifier。
2. **生命周期标记不符合新设计。** `MarkRemoveAfterBattle()` 只能保证战斗结束清除，不能满足“本回合结束清除”。如果抽到的是保留牌或进入弃牌/出牌区后又回流，附魔会跨回合存在。

此外，用户明确要求“先结算回合结束附魔，再移除临时添加附魔”，所以不能简单在 `TriggerOnTurnEnd` 之前删除；应在所有 OnTurnEnd 类效果结算之后、牌被丢弃/保留迁移之前或之后的明确节点统一清理。

### 推荐方案

**方案 A：新增“一回合临时附魔”生命周期（推荐）**

1. 在 `MaterialModifierModel` 增加独立标记，例如 `RemoveModifierAfterTurn`，以及方法 `MarkRemoveAfterTurn()`。
2. 在 `MaterialModel` 增加 `RemoveTurnOnlyModifiers()`，从 `modifiers` 中移除该标记的附魔并 `RebuildModifierFlags()`。
3. 在 `PlayerState` 增加 `RemoveTurnOnlyModifiers()`，覆盖 `Hand`、`PlayZone`、`DrawPile`、`DiscardPile`、`ConsumedPile`、`TemporaryMaterialsNextTurn`；是否覆盖 `Deck` 需要谨慎，通常运行时战斗克隆牌可覆盖，永久牌库不应被误清。
4. 把 `VortexNextDrawBuffModel`、`KindlingNextDrawBuffModel`、`ChargeNextDrawBuffModel` 中这类“下次抽到的牌获得附魔”的新增附魔从 `MarkRemoveAfterBattle()` 改为 `MarkRemoveAfterTurn()`，除非文案明确写“本场战斗中添加”。
5. 在 `BattleManager.EndPlayerTurnRules()` 中按用户要求放置清理点：
   - 先 `PlayerState.TriggerOnTurnEnd(...)`。
   - 再 `TriggerMagicTurnEnd()`。
   - 然后 `PlayerState.RemoveTurnOnlyModifiers()`。
   - 再 `PlayerState.EndTurn(removedTemporaryCards)`，处理丢弃/保留/临时牌移除。
6. 保留 `RemoveBattleOnlyModifiers()` 用于“本场战斗中添加”的附魔，两套生命周期互不替代。

**方案 B：只在三个 Buff 中防重复并改成弃牌时移除（不推荐）**

可以临时让涡流/电荷/引燃在 `OnDiscard` 时移除自己，但这会依赖牌是否被丢弃：

- 保留牌不丢弃时可能继续留存。
- 出牌区、抽牌堆、弃牌堆之间的移动路径复杂，容易漏掉。
- 不满足“回合结束统一先结算后移除”的设计表达。

因此建议只作为热修补丁，不作为最终方案。

**防重复细节**

- `VortexNextDrawBuffModel` 添加前检查 `card.modifiers` 是否已有 `VortexModifier`。
- `ChargeNextDrawBuffModel` 添加前检查是否已有 `ChargeModifier`。
- `KindlingNextDrawBuffModel` 已有防重，可保留并只切换生命周期。
- 无论目标已有同类附魔与否，是否消耗 1 层 Buff 需要你确认。按当前 `KindlingNextDraw` 的实现，已有附魔也会 `ConsumeStack(1)`；建议三者保持一致，避免无限寻找下一张无附魔牌。

### 风险

- 新增生命周期会影响所有被标为 `RemoveModifierAfterTurn` 的附魔，需要严格只迁移这三类“下次抽牌添加”的附魔，避免误删保留/永久/战斗内附魔。
- 清理点如果放在 `EndTurn()` 后，保留牌和已进入弃牌堆的牌都能清到，但部分 `OnDiscard` 逻辑可能已带着临时附魔执行；用户要求先结算回合结束附魔，不等于要求弃牌触发也保留临时附魔。推荐放在 `TriggerMagicTurnEnd()` 后、`EndTurn()` 前。
- 如果未来某个临时附魔本身拥有 `OnTurnEnd`，该顺序可保证它先触发再被清理。

### 验收

1. 已有漩涡的牌再次被 `VortexNextDraw` 命中时，不新增第二个漩涡词条。
2. 本回合抽到并获得漩涡/电荷/引燃的牌，在回合结束所有 OnTurnEnd 效果结算后移除对应附魔。
3. 保留在手牌的牌不会把这类临时附魔带到下一回合。
4. 明确“本场战斗中添加”的附魔仍保留到战斗结束，不被回合结束清理。
5. 战斗结束仍会清理 `RemoveModifierAfterBattle` 和临时战斗牌，不影响旧逻辑。


### 审阅意见
采用方案A

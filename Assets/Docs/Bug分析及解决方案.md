# Bug分析及解决方案

> 基于 `Assets/Docs/待修复Bugs.md` 与当前项目代码结构整理。
> 本文先以“程序逻辑”问题为主，UI / 布局 / 美术 / 文本适配类问题先做定位与低优先级建议，不在这一轮深入到视觉细调方案。
> 分类采用三类：程序逻辑、素材加载、文本适配。对于纯表现/UI问题，若不属于后两类，暂归入“程序逻辑（UI表现层）”，便于后续统一排期。

## 总体结论

### 优先级建议
- 第一优先：会直接导致规则错误、存档错误、事件错误、战斗结算错误的问题。
- 第二优先：会造成玩家误解、但未必破坏规则闭环的问题。
- 第三优先：UI、布局、美术、字号、动画表现类问题。

### 当前框架观察
- 项目当前战斗系统的“箭头读取”和“事件配方匹配”是两套体系：
  - 战斗读取走 `ArrowReadToken` / `ArrowReadStep`。
  - 事件配方匹配直接按 `MaterialModel` 卡牌张数匹配。
- 这会导致“双重箭头 / 四重箭头 / 脆弱箭头 / 特殊箭头修饰”在战斗与事件中的语义不一致。若后续要彻底稳定扩展，事件系统更适合复用现有箭头读取结果，而不是继续按“卡牌张数”单独判断。
- 当前商店状态没有进入 `RunSaveData`，说明“进入商店后退出再恢复原样”目前不是漏写某一行，而是存档结构本身未覆盖商店状态。
- 当前很多 modifier 的触发语义依赖 `OnJoin / OnInvoke / OnEnd / OnTurnStart` 等通用钩子，这本身是好框架；问题主要出在个别效果的触发时机与设计预期不一致，不建议为单个道具新增专用接口，优先在现有钩子/读取链路内修。

## 逐条分析

## 1. 具有电荷的箭头似乎可以通过反复进出出牌区，给对面加电弧效果
- 类型：程序逻辑
- 现象定位：`Assets/Scripts/Modifiers/ChargeModifier.cs`
- 问题位置：`ChargeModifier.OnJoin()` 每次进入出牌区都会调用 `BattleManager.AddArcToRandomEnemy(1)`。
- 触发条件：同一张带电荷的箭头，被玩家反复从手牌拖入出牌区，再从出牌区撤回手牌，再次拖入。
- 根因分析：当前效果绑定在 `OnJoin`，但没有“本次出牌是否已触发”的去重状态；`TryMovePlayCardToHand` 撤回后也不会消费/锁定该触发。
- 解决方案：
  - 方案A：将该效果限制为“该牌本回合第一次进入出牌区时触发”，在 `MaterialModel` 的战斗临时状态里记录一次性标记。
  - 方案B：若设计语义其实是“读取时触发”或“打出并结算时触发”，则应把效果从 `OnJoin` 挪到更合适的现有钩子（如 `OnInvoke` 或读取结算链路）。
- 推荐：优先方案A，改动最小，仍复用现有 modifier 钩子体系。
- 复杂度：低
- 对扩展性影响：低。只要把“入场一次性触发”做成通用战斗临时标记，不会破坏框架。
- 风险：中。需要先确认“电荷”的正式设计语义，否则可能修成“不会刷电弧”，但也顺手改掉了原本想保留的拖入触发体验。
- 审阅意见：类似于方案B，这里应该是指结算时触发（包括箭头基础效果结算和法术匹配结算），请修复

## 2. 具有脆弱的箭头似乎无法被读取配方，也没有基础效果
- 类型：程序逻辑
- 现象定位：`Assets/Scripts/Modifiers/FragileArrowModifier.cs`、`Assets/Scripts/Data/ArrowReadSystem.cs`
- 问题位置：`FragileArrowModifier.GetArrowMatchTokenCount()` 直接返回 `0`。
- 触发条件：任意带脆弱 modifier 的箭头进入读取序列时。
- 根因分析：战斗配方匹配依赖 token 数；该 modifier 把 token 数压成 0，导致此箭头不参与读取。与此同时，它的 `AfterReadAction` 是 `SplitIntoHalfArrowsToDiscard`，说明本意更像“读完后裂成半箭头并进入弃牌”，而不是“完全不被读取”。
- 解决方案：
  - 方案A：让脆弱箭头保留至少 1 个匹配 token，读取后再走 `SplitIntoHalfArrowsToDiscard`。
  - 方案B：若设计语义真的是“不计入配方、只保留某种基础效果”，则需要明确该基础效果应在哪个钩子执行；但当前代码里看不到这样的补偿逻辑。
- 推荐：优先方案A。因为它最符合现有 `AfterReadAction` 语义，也不需要新接口。
- 复杂度：低
- 对扩展性影响：低。是在现有读取框架内修 token 计数，不会增加特判接口。
- 风险：低
- 审阅意见：选择方案A，并且分裂行为应该出现在所有序列读取完之后。（按时序应该会在所有打出的素材箭头读取完之后，有一个结算所有箭头变化的演出环节，结算完之后才会进入弃牌环境）

## 7. 四重上第一个打出，火酒只触发了两次
- 类型：程序逻辑
- 当前定位：`Assets/Scripts/Modifiers/BigArrow4Modifier.cs`、`Assets/Scripts/HandSystemUI.cs:3678`、`Assets/Scripts/HandSystemUI.cs:3899`
- 问题位置：四重箭头只增加 token 数；法术匹配逻辑从 `startIndex` 开始做一次 `magic.IsMatch(tokens, startIndex)`，不是在同一个多 token step 上做完整滑窗匹配。
- 触发条件：前置是一个提供多个 token 的箭头（如四重箭头），且法术是长度较短、理论上可在这串 token 内重复触发的配方。
- 根因分析：当前读取框架更像“从当前 token 起，找能匹配的一组法术”，而不是“在该多重 token 序列内逐个滑窗尝试所有重叠组合”。因此长度为 4 的上箭头，对长度 2 的“火酒”只可能被部分覆盖，出现少触发。
- 解决方案：
  - 方案A：明确规则，规定多重箭头只是“扩充连续 token 序列”，但同一法术只按读取步进触发当前允许次数。
  - 方案B：修改读取/施法算法，使多 token 来源支持滑窗式连续匹配。
- 推荐：先确认设计。如果用户预期“四重上”确实应触发 3 次长度 2 配方，那么必须改算法，不能只补特判。
- 复杂度：中到高
- 对扩展性影响：
  - 若改为通用滑窗读取，扩展性更强。
  - 若只给“火酒 + 四重箭头”加特判，扩展性极差，不可接受。
- 风险：高。会影响所有多重箭头与短配方法术的结算语义。
- 审阅意见：确认需要修复，应该在玩家确认结束回合后，将出牌区的内容转化为实际的素材Token（注意，虽然变为了token，但演出动画还是在原本的出牌区的，所以需要对每个token记录其对应的出牌区箭头，方便演出？）

## 8. 卡牌附魔的漩涡好像没起效
- 类型：程序逻辑
- 当前定位：`Assets/Scripts/Modifiers/VortexModifier.cs`
- 问题位置：`OnJoin()` 给随机敌人上 debuff，`OnEnd()` 抽 1 张到出牌区。
- 根因分析：代码本身不是“没效果”，而是效果触发时机可能与玩家理解不一致。如果玩家认为“漩涡”应该在读取时/施法时触发，那么当前实现就会被感知为没起效。
- 解决方案：
  - 先核对设计文案与实际钩子是否一致。
  - 若设计确实要求“读取时触发”，则应把效果迁回现有读取结算钩子，不要新增单独接口。
- 复杂度：低到中
- 对扩展性影响：低。核心是修正挂钩位置。
- 风险：中。先确认设计语义再动。
- 审阅建议：“触发道具”指以该箭头产生的Token为素材之一触发道具的时候。（如果将素材遍历变为Token遍历，也许需要在触发时去获取Token所属箭头素材的附魔效果？）

## 9. 假钻石移除的牌好像没进消耗堆，而且移除好像是永久的
- 类型：程序逻辑
- 当前定位：商店/事件删牌流程与战斗消耗流程分离；战斗消耗入口为 `PlayerState.ConsumeCardForBattle()`。
- 根因分析：当前项目里“消耗堆”是战斗内临时区；而“删牌”是直接改 `Deck` 的持久结果。如果“假钻石”复用了删牌流程，那么它表现成“永久移除且不进消耗堆”是合理的代码结果，但可能不符合设计。
- 解决方案：
  - 先确认“假钻石”的设计语义：
    - 若应该是“本战斗消耗”，应改走战斗消耗链。
    - 若应该是“永久删牌”，那现在不是 bug，而是表现说明不足。
- 复杂度：中
- 对扩展性影响：中。关键是要复用现有“删牌 / 消耗 / 临时移除”三套语义之一，不能再做第四套接口。
- 风险：高。因为这条实际可能是设计歧义，不是纯代码错。
- 审阅建议：是战斗消耗。所有“消耗”字样都是指进入消耗堆（这种描述通常只会负责战斗中效果出现），如果是一次性卡牌，通常会有明显描述（如：从本局游戏中移除）。


## 10. 苍蝇的换牌有 bug：现在有回合结束时清除，但是实际上换牌生效应该在下回合，但是打出的这回合就被清除了
- 类型：程序逻辑
- 当前定位：下回合生效类效果应依赖 `OnTurnStart` / `NextDraw` / `TemporaryMaterialsNextTurn` 机制，而非直接改当前手牌实例。
- 根因分析：很可能当前效果把“换牌结果”直接挂在当前回合手牌对象或本回合临时状态上，导致 `EndTurn` 后清理掉；而正确语义应在“下回合抽牌/发牌阶段”注入。
- 解决方案：
  - 把该效果改为记录在已有的“下回合生效”容器里（如 next-turn buff、临时牌列表、抽牌后附魔钩子），不要直接附着在当回合将被清理的牌实例上。
- 复杂度：中
- 对扩展性影响：低。属于回归现有 next-turn 框架，不是新增接口。
- 风险：中。需要先定位苍蝇具体敌人/效果数据挂载点。
- 审阅建议：应该是下回合生效，你可以制作一个“下回合抽牌”的临时buff，效果是在下回合开始重置换牌次数时，对次数进行修正，然后这个Buff会在修正结束（即玩家回合开始阶段结束后消失）。然后我还需要你去检查一下所有描述是“下回合生效”的道具与buff，检查它们是否会存在触发道具/获取buff后，buff在本回合结束（结算结束）后就消失的情况，并予以修正。

## 24. 事件提供生命上限的同时，也增加了血量
